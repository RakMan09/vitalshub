Your health data, unified & owned.

VitalsHub pulls your data out of mismatched sources — a smartwatch export, a smart-scale CSV, an Apple Health dump, and a clinic's HL7 v2 message — and unifies them into one clean, validated FHIR timeline. Then it lets you share a de-identified snapshot under consent and logs exactly who saw what.

How it works

flowchart LR
  W["Wearable JSON"] --> Ing
  S["Smart-scale CSV"] --> Ing
  H["Clinic HL7 v2"] --> Ing
  A["Apple-Health XML"] --> Ing
  Ing["Adapters"] --> Norm["Normalize to FHIR"]
  Norm --> Term["Terminology map (LOINC/SNOMED)"]
  Term --> Valid["Validate"]
  Valid -->|valid| Store[("Resource store")]
  Valid -->|invalid| Quar[("Quarantine + reason")]
  Store --> Deid["De-identify + consent"]
  Deid --> Doc["Doctor / coach"]
  Store -.every read/write.-> Audit[("Audit log")]

Standards-based

Parses HL7 v2, produces FHIR R4 resources — the formats healthcare actually runs on.

Terminology mapping

Local codes → LOINC/SNOMED. Unmapped codes are flagged, never silently dropped.

PHI de-identification

Safe-Harbor redaction: names, contacts, addresses, IDs removed; dates → year; ages 90+ aggregated.

Un-bypassable audit

One door to the database, and it always writes a FHIR AuditEvent. Proven at 100% coverage by tests.

How do you know it really works?

  • Automated tests — the backend ships 17 passing tests (adapters, unification, quarantine, de-id, consent, audit coverage). See the CI badge on the repo.
  • This demo — the exact same pipeline logic runs here in your browser; its outputs are checked by a Node self-test in CI.
  • Run it yourself — ./gradlew bootRun then python data/synthea_load.py reproduces the whole flow against the real API.
  • API explorer — the running app serves interactive Swagger UI at /swagger-ui.html and a FHIR CapabilityStatement at /fhir/metadata.
